### [A] DB 인덱스 동작 방식 및 최적화

#### 기본 질문
“DB 인덱스가 어떻게 동작하는지 설명해주시고, 인덱스를 사용할 때 발생하는 쓰기 성능 저하나 공간 사용 등의 trade-off를 어떻게 해결하셨는지 말씀해 주세요.”

#### 예시 답변
> “DB 인덱스는 테이블 내 데이터를 빠르게 검색하기 위한 보조 자료구조입니다. 일반적으로 B-Tree 구조를 사용하며, 이 구조는 데이터가 정렬된 상태로 저장되어 각 노드마다 여러 키를 한 번에 비교할 수 있습니다. 이로 인해 검색 시 평균 O(log n)의 시간 복잡도를 달성할 수 있습니다.
> 
> 다만, 인덱스는 삽입, 업데이트, 삭제 시 추가적인 작업을 요구하여 쓰기 성능에 영향을 줄 수 있고, 디스크 공간을 추가로 사용합니다. 따라서 테이블의 사용 패턴과 쿼리 분석을 통해 반드시 필요한 컬럼에만 인덱스를 적용하거나, 파티셔닝 기법을 활용해 부하를 분산하는 등 최적화 전략을 사용합니다.”

#### 추가(꼬리) 질문 및 상세 답변
- **추가 질문:**

- **예시 답변:**  
  > “카탈로그 서비스 프로젝트에서, 상품 데이터가 빈번하게 업데이트되면서 특정 인덱스가 과도한 유지 비용을 발생시켰습니다. 이를 해결하기 위해 해당 컬럼의 쿼리 빈도와 업데이트 패턴을 면밀히 분석한 후, 불필요한 인덱스를 제거하고, 꼭 필요한 인덱스의 경우에는 재구성을 진행했습니다. 주기적인 인덱스 재빌드 작업과 실시간 모니터링을 도입하여 인덱스 관련 성능 저하를 사전에 예방할 수 있었습니다.”

### [B] JVM 및 메모리 관리

#### 기본 질문
“JVM의 내부 구조와 각 메모리 영역(Heap, Stack, Metaspace 등)의 역할, 그리고 주요 가비지 컬렉션(GC) 알고리즘에 대해 설명해 주세요.”

#### 예시 답변
> “JVM은 크게 **클래스 로더**, **실행 엔진**, **메모리 관리** 모듈로 구성됩니다.
> 
> - **클래스 로더:**  
>   - 애플리케이션에서 사용하는 클래스를 읽어 JVM 메모리에 적재하며, 로딩, 연결, 초기화 단계를 거칩니다.
> 
> - **메모리 영역:**  
>   - **Heap:**  
>     - 객체가 생성되는 메모리 영역으로, 가비지 컬렉션의 주요 대상입니다.
>   - **Stack:**  
>     - 각 스레드마다 존재하며, 메서드 호출 시 생성되는 프레임과 지역 변수가 저장됩니다.
>   - **Metaspace:**  
>     - 클래스 메타데이터가 저장되는 영역으로, 예전의 PermGen을 대체합니다.
> 
> - **실행 엔진:**  
>   - 바이트코드를 기계어로 변환해 실행하며, JIT(Just-In-Time) 컴파일러를 통해 실행 성능을 최적화합니다.
> 
> - **가비지 컬렉션(GC):**  
>   - JVM은 Heap에서 더 이상 참조되지 않는 객체를 자동으로 제거합니다.  
>   - 주요 알고리즘으로는 Mark-Sweep, Mark-Compact, 그리고 Generational GC가 있습니다.  
>   - 특히 Generational GC는 객체의 생존 기간에 따라 Young Generation과 Old Generation으로 나누어 관리하여, 대부분 단기간 존재하는 객체를 빠르게 청소함으로써 전체 성능을 향상시킵니다.”

#### 추가(꼬리) 질문 및 상세 답변
- **추가 질문:**  
  “실제 운영 중에 GC 튜닝이나 메모리 누수 문제를 해결한 경험이 있으신가요? 구체적인 원인 분석과 해결 방법에 대해 설명해 주세요.”
- **예시 답변:**  
  > “프로젝트 초기 단계에서 메모리 누수가 발생해 GC 로그를 분석한 결과, 특정 라이브러리의 캐싱 정책에 문제가 있음을 확인했습니다. 해당 라이브러리의 만료 정책을 재설계하고, 불필요한 객체 참조를 줄이도록 코드를 개선했습니다. 또한, Young Generation과 Old Generation의 크기 비율을 조정하는 등 GC 튜닝을 진행해 애플리케이션 응답 시간을 안정화한 경험이 있습니다.”

### [C] 동시성 제어: ConcurrentHashMap 및 Atomic 계열

#### 기본 질문 (ConcurrentHashMap)

#### 예시 답변 (ConcurrentHashMap)
> “ConcurrentHashMap은 멀티스레드 환경에서 안전하게 데이터를 읽고 쓸 수 있도록 설계된 자료구조입니다.  
> - **Java 7 이하:** 내부를 여러 세그먼트로 분할하여 각 세그먼트별로 별도의 락을 사용했습니다.  
> - **Java 8 이후:** 세그먼트 개념을 제거하고, 노드 기반의 자료구조와 트리 구조를 결합하여 읽기 작업은 락 없이 처리하고, 쓰기 작업은 최소 범위의 synchronized 블록과 CAS(Compare-And-Swap) 연산을 사용해 동시성을 보장합니다.
> 이로 인해 다수의 스레드가 동시에 접근하더라도 데이터 무결성을 유지하면서 높은 성능을 발휘할 수 있습니다.”

- **예시 답변:**  
  > “한 프로젝트에서 여러 스레드가 동시에 데이터를 업데이트하는 환경에서 기존 HashMap 사용 시 데이터 손실이 발생했습니다. ConcurrentHashMap으로 전환한 후, 데이터 업데이트 로직에 CAS 연산을 도입해 락 경합을 최소화했습니다. 결과적으로, 다수의 스레드가 동시에 읽기 및 쓰기 작업을 수행하더라도 데이터 무결성이 보장되고, 성능 저하 없이 안정적인 시스템 동작을 확인할 수 있었습니다.”

#### 기본 질문 (Atomic 계열 vs Concurrent 컬렉션)
“concurrent와 atomic 함수가 있는데 뭐가 달라?”

#### 예시 답변
> **Atomic 계열:**  
> - **용도:** 단일 변수에 대해 원자적인 연산(읽기, 쓰기, 증감 등)을 수행합니다.  
> - **구현:** 내부적으로 CAS(Compare-And-Swap) 연산을 사용해 락 없이 원자적 연산을 보장합니다.  
> - **적용 예:** 카운터, 플래그, 단일 변수 업데이트 등 단순한 상태 업데이트에 적합합니다.
>
> **Concurrent 컬렉션:**  
> - **용도:** 여러 요소를 담는 자료구조의 동시 접근 문제를 해결합니다.  
> - **구현:** 내부적으로 부분적인 락, CAS 또는 다른 동시성 제어 기법을 사용하여, 여러 스레드가 동시에 데이터에 접근하거나 수정해도 안전하게 작동합니다.  
> - **적용 예:** ConcurrentHashMap, CopyOnWriteArrayList 등 복잡한 자료구조에 대해 동시성 문제를 해결할 때 사용합니다.
>
> **요약:**  
> Atomic 계열은 단일 변수의 원자적 연산에 초점을 맞추는 반면, Concurrent 컬렉션은 복합 자료구조의 동시 접근을 안전하게 처리하는 데 중점을 둡니다.

### [D] 트랜잭션 관리 및 격리성

#### 기본 질문
“트랜잭션이 무엇이며, ACID 원칙과 트랜잭션 관리의 중요성에 대해 설명해 주세요.”

#### 예시 답변
> “트랜잭션은 데이터베이스에서 하나의 작업 단위로, 여러 데이터 조작 작업을 하나의 논리적 단위로 묶어 실행하는 것을 의미합니다. 트랜잭션은 **ACID 원칙**을 준수해야 합니다:
> - **Atomicity (원자성):** 모든 작업이 전부 실행되거나 전혀 실행되지 않아야 합니다.
> - **Consistency (일관성):** 트랜잭션 실행 전후에 데이터베이스는 항상 일관된 상태를 유지해야 합니다.
> - **Isolation (격리성):** 동시에 실행되는 트랜잭션들이 서로 간섭하지 않고, 독립적으로 처리되어야 합니다.
> - **Durability (내구성):** 트랜잭션이 성공적으로 완료되면 그 결과는 영구적으로 저장되어야 합니다.
> 
> 트랜잭션 관리는 데이터 무결성과 안정성을 보장하기 위해 필수적이며, 특히 복잡한 비즈니스 로직이나 분산 시스템에서 중요한 역할을 합니다.”

#### 추가(꼬리) 질문 및 상세 답변 (격리성 포함)
- **추가 질문:**  
  “트랜잭션 격리성에 대해 설명해 주시고, 실제 프로젝트에서 격리성 수준을 조정하여 발생한 문제를 해결한 경험이 있으시다면 구체적으로 말씀해 주세요.”
- **예시 답변:**  
  > “트랜잭션 격리성은 여러 트랜잭션이 동시에 실행될 때 서로의 작업이 영향을 주지 않도록 하는 보장 수준입니다. 대표적인 격리 수준은 다음과 같습니다:
  > - **Read Uncommitted:** 다른 트랜잭션의 커밋되지 않은 데이터를 읽을 수 있어 '더러운 읽기'의 문제가 발생할 수 있습니다.
  > - **Read Committed:** 커밋된 데이터만 읽으므로 '더러운 읽기'는 방지하지만, 반복 읽기나 팬텀 리드(Phantom Read) 문제는 발생할 수 있습니다.
  > - **Repeatable Read:** 한 트랜잭션 내에서 같은 데이터를 여러 번 읽어도 일관된 결과를 보장하지만, 팬텀 리드가 발생할 수 있습니다.
  > - **Serializable:** 가장 높은 격리 수준으로, 트랜잭션을 순차적으로 실행하는 것과 동일한 효과를 내어 모든 동시성 문제가 해결되지만, 성능 저하가 발생할 수 있습니다.
  > 
  > 한 프로젝트에서는 동시 업데이트로 인한 데이터 정합성 문제가 발생하여 기본 격리 수준인 Read Committed를 사용하던 중 팬텀 리드가 문제가 되었습니다. 이에 Repeatable Read로 격리 수준을 높여 동일 트랜잭션 내에서 읽은 데이터의 일관성을 보장하고, 동시에 성능 저하를 최소화하는 최적의 설정을 찾았습니다.”
